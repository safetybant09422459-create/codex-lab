# Gift Skill v0.1（2026-07-14）

## 実施日時

2026-07-14（Asia/Tokyo）

## 変更内容

Gift SkillのProvider、Tool、Runtime登録、Repository、SQLite Storage、migration、Consumer API、Web UI、tests、docsを追加した。
候補・贈った・もらった記録を、人、日付、任意金額、メモ、イベント、記念日と共に保存できる。

## 変更理由と設計判断

候補Skillを日常頻度、最小実装の有用性、長期蓄積価値、外部依存、プライバシーリスクで比較した。
Meal/Shoppingは頻度が高いが、共有在庫と通知までないと既存メモとの差が小さい。Calendarは外部同期と競合解決が先に必要。
Giftは小さな正本から価値を出せ、履歴が年々家族固有の資産になるため採用した。

DBはTravelと分離した`gift.db`とし、versioned migrationを正本にした。Providerは事実のCRUDとfilterだけを担当し、
「何を贈るべきか」の意味判断はLLMへ残した。writeは既存RuntimeのPermission / Confirmation / Auditを迂回しない。

## 参考資料

* Apple Remindersのtag / Smart Listによる高速な分類とfilter
* Google Keepの共有ノートとcollaboratorモデル
* Jarvis Skill Standard Architecture、Provider Principle、Observation Guardrail

これらのUIをコピーせず、Gift固有のgiver / recipient / occasionを正規fieldとして持つ設計にした。

## 代替案

* Meal/Shopping: 次候補。Inventoryとhousehold共有契約を先に設計する。
* Calendar: 高価値だが、CalDAV/Google/Apple同期と競合ポリシーが必要。
* 汎用NoteにGift tagを付ける: 実装は小さいが、自然言語質問に必要なgiver/recipient/dateの正本性が弱い。
* AIが自由文から自動保存: Trust Before Automationに反するため不採用。

## データ保全と影響範囲

変更前に既存`storage/travel.db`を実体表示せず`/tmp/jarvis-travel-before-gift-20260714.db`へbackupし、SHA-256一致を確認した。
既存DB schemaは変更していない。新規`gift.db`だけを作成する。外部API、購入、通知、service操作はない。

## ロールバック

新規Gift route/module/Provider/Tool/Skillを外し、`gift.db`を退避すれば既存Skillへ影響せず戻せる。
データ削除は行わず、必要ならDBを保持したまま機能だけ無効化する。

## 残る制約

* household/member認証とsurprise visibilityは未実装。
* Web writeの確認はConfirmation Transactionではなくローカル環境向け暫定方式。
* edit/archive、写真、Calendar ID、通知、購入連携は未実装。
* 金額はv0.1では日本円整数。multi-currencyは将来migrationで追加する。
